# PHP Laravel 新手入門教案綱要

> 以本 Repo「書籍管理系統」為實作主軸，從零帶出 Laravel 全棧開發核心技能。
>
> **給教案作者的提醒**：本綱要為「骨架級」指導文件，內容刻意詳盡以確保教案撰寫時不遺漏關鍵要素。依此綱要寫出的**教案本身應力求精練**——每個單元的講義以「此刻實作所需的最小可行知識」為上限，避免資訊過載。
>
> **單元排列原則**：遵循「對應專案流程的漸進式推演」——教案對應專案推進流程（需求分析 → 討論 → 拆解 → 規劃 → 設計 → 開發 → 測試），在開發階段以「可視性」為脈絡，先讓學員看到畫面（假資料），再串接真實資料庫完成 CRUD 循環，最後才抽出 Service/Resource 分層與進階功能。每一步都有可見的產出，避免「寫了半天什麼都看不到」的挫折。

## §2 學習成果（結訓時能做到）

1. **能說明** Laravel 分層架構（Route → Controller → Service → Repository → Model）各層的職責與呼叫方向
2. **能使用** Composer 建置專案、用 Docker Compose 一鍵啟動整套開發環境（App + MySQL + Redis）
3. **能撰寫** Laravel Migration 腳本管理資料庫 schema 演進，並解釋版本控管與回滾機制
4. **能實作** Eloquent Model 與自訂查詢方法，理解 Active Record 模式與關聯定義
5. **能設計** RESTful API（正確使用 HTTP 動詞與狀態碼），用 Form Request + Exception Handler 統一驗證與錯誤回應，並產出 Swagger 互動式文件
6. **能製作** Blade 伺服器端渲染頁面（列表、表單、詳情），理解 PRG 模式
7. **能實作** Redis Cache-Aside 快取模式，並用 redis-cli 驗證快取行為
8. **能撰寫** PHPUnit/Pest 單元測試與 Feature Test，達到基本覆蓋率門檻

### U01｜專案啟動與第一支 API（90 min）

#### ① 為什麼先教這個？

讓學員在第一堂課就看到「程式跑起來」的成就感。沒有什麼比親手讓一個 Web 應用回應你的請求更能建立學習信心。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `composer.json` | 依賴管理與建置設定（定義專案用到的所有套件） |
| `bootstrap/app.php` | Laravel 應用程式進入點 |
| `routes/web.php` | Web 路由定義 |
| `Http/Controllers/HelloController.php` | 最簡易的 Controller，示範如何回應 HTTP 請求 |

#### ③ 核心觀念

- **專案標準目錄架構** — `app/`（程式碼）、`config/`（設定）、`resources/`（模板與前端）、`routes/`（路由）、`tests/`（測試），各司其職
- **Composer 做了什麼** — 下載依賴、自動載入類別；`composer install` 與 `composer update` 的差異
- **`bootstrap/app.php`** — Laravel 12 的應用程式配置入口，取代舊版 Kernel
- **`Route::get()` 路由定義** — 把 URL 對應到 Controller 方法
- **內嵌開發伺服器** — `php artisan serve` 就能跑，不用另外裝 Apache/Nginx

#### ④ 動手做

**必做**
1. 執行 `docker compose up -d` 啟動 MySQL 與 Redis，再執行 `php artisan serve`，瀏覽器打開 `http://localhost:8000/hello`，確認看到回應
2. 在 `routes/web.php` 新增一個 `/whoami` 路由，對應到 HelloController 的 whoami 方法，回傳自己的名字（JSON 格式）

**延伸挑戰**
1. 觀察 `composer.json` 的 require 區塊，寫下每個套件的用途

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| `artisan serve` 後報 port 衝突 | Port 8000 被其他程式占用 | macOS：`lsof -i :8000` 找出占用程式並關閉；或 `php artisan serve --port=8080` |
| 修改 PHP 後瀏覽器沒變化 | 瀏覽器有快取或忘了儲存 | Ctrl+Shift+R 強制重新整理；確認檔案已儲存 |
| Class not found | autoload 未更新 | `composer dump-autoload` |

#### ⑥ 驗收標準（DoD）

能用瀏覽器或 `curl http://localhost:8000/whoami` 呼叫自訂端點並看到正確的 JSON 回應。

### U02｜假畫面 — Blade 初體驗（60 min）

#### ① 為什麼先教這個？

U01 讓程式跑起來了，但只有文字回應。這堂課讓學員第一次「看到畫面」——用 Blade 做出一個有書籍清單的 HTML 頁面。資料先用 Controller 中的假資料（hardcoded），不需要 DB，**重點是讓學員體驗「改了模板，刷新就看到結果」的即時回饋感**。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Http/Controllers/BookController.php` | MVC 端點（回傳 HTML 頁面），此刻先用假資料 |
| `views/books/index.blade.php` | 書籍清單頁 |
| `views/books/show.blade.php` | 書籍詳情頁 |
| `views/layouts/app.blade.php` | 共用 Layout |

#### ③ 核心觀念

- **伺服器端渲染（SSR）** — Blade 在 server 端組好完整 HTML 再送出，不需要前端框架
- **`return view()`** — Controller 回傳 `view('books.index')` 代表渲染 `views/books/index.blade.php`
- **`compact()` 或 `with()`** — 把資料從 Controller 傳給模板
- **Blade 關鍵語法**：
  - `@foreach` — 迴圈渲染（類似 for-each）
  - `{{ $var }}` — 輸出文字（自動 XSS 防護）
  - `@extends` / `@section` / `@yield` — Layout 繼承
- **此刻不需要 DB** — Controller 直接建立假的書籍清單（`collect([...])`）丟給模板，先專注學 Blade 語法

> **講師提示**：先 demo 假資料版本，讓學員專心學 Blade 語法。後面 U06 串接 DB 時，只要改 Controller 的資料來源，頁面不用動——這就是模板引擎的價值。

#### ④ 動手做

**必做**
1. 觀察 `BookController@index` 如何把資料傳給 view，對照 `index.blade.php` 中的 `@foreach` 如何渲染每一行
2. 寫一個簡易版 Controller 方法，回傳一個包含 3 本假書的清單，瀏覽 `http://localhost:8000/books` 看到表格
3. 點進某本書的連結，觀察詳情頁如何用 `{{ }}` 顯示各欄位

**延伸挑戰**
1. 修改 `index.blade.php`，在表格加一欄「狀態」，當庫存 = 0 時顯示紅字「缺貨」（提示：`@if` + inline style）

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| View not found | Controller 回傳的 view 名稱錯誤（如 `book.list` 而非 `books.index`） | 確認 `views/` 下的資料夾名稱與 view 名稱完全一致 |
| 頁面顯示 `{{ $books }}` 原始文字 | 忘了用 `.blade.php` 副檔名 | Blade 檔案必須是 `.blade.php` 結尾 |
| Layout 沒套用 | 忘了 `@extends('layouts.app')` | 在 Blade 檔案開頭加上 extends |

#### ⑥ 驗收標準（DoD）

能在瀏覽器看到一個有書籍清單的 HTML 頁面（即使資料是假的），並能說出 `@foreach` 和 `{{ }}` 的用途。

### U03｜API 回傳假資料（60 min）

#### ① 為什麼先教這個？

U02 做了給人看的頁面，這堂課做給機器用的 API。同樣用假資料，讓學員先理解「Controller 怎麼回傳 JSON」。有了 API，就能用 Swagger UI 互動式測試，也為後面串接真實 DB 打下基礎。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Http/Controllers/Api/BookApiController.php` | REST API 端點（回傳 JSON），此刻先用假資料 |
| `routes/api.php` | API 路由定義 |

#### ③ 核心觀念

- **`return response()->json()`** — 方法回傳值序列化為 JSON，與 U02 的 `view()` 對比
- **`routes/api.php`** — API 路由自動加上 `/api` 前綴
- **RESTful 動詞對應** — GET（查）、POST（建）、PUT（改）、DELETE（刪）
- **Swagger UI** — 訪問 `/api/documentation` 可互動式測試所有 API，無需額外工具
- **此刻不需要 DB** — Controller 直接回傳假的陣列，先專注學 REST 概念

#### ④ 動手做

**必做**
1. 寫一個 `GET /api/books` 端點，回傳包含 3 本假書的 JSON 陣列
2. 寫一個 `GET /api/books/{id}` 端點，依 ID 回傳單筆假書（固定回傳同一本即可）
3. 打開 Swagger UI（`http://localhost:8000/api/documentation`），測試剛才寫的 API

**延伸挑戰**
1. 加入 `POST /api/books` 端點，接收 JSON body 並用 `dd()` 印出收到的資料（還沒有 DB，先確認能收到請求）

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| API 回 404 | 路由定義在 `web.php` 而非 `api.php` | API 路由應放在 `routes/api.php` |
| Swagger UI 看不到新加的 API | 沒有加上 OpenAPI 註解 | 加上 `@OA\Get` 等註解並執行 `php artisan l5-swagger:generate` |

#### ⑥ 驗收標準（DoD）

能用 Swagger UI 或 `curl http://localhost:8000/api/books` 看到 JSON 回應，並能說出 API 與 MVC 頁面的差異。

### U04｜設定檔與多環境切換（60 min）

#### ① 為什麼先教這個？

接下來要串接真實 DB（U05-U06），得先搞懂「DB 連線設定在哪、怎麼改、怎麼切環境」。不懂設定管理，後面 MySQL 連不上就會卡住。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `.env` | 環境變數檔（本機開發用） |
| `.env.example` | 環境變數範本 |
| `config/database.php` | 資料庫連線設定 |
| `docker-compose.yml` | 多容器編排（App + MySQL + Redis） |

#### ③ 核心觀念

- **`.env` 環境變數** — 敏感資訊（密碼、API Key）不進版控，用 `.env` 管理
- **`config()` 讀取設定** — `config('database.default')` 讀取 `config/database.php` 的值
- **`env()` 讀取環境變數** — `env('DB_HOST', 'localhost')` 讀取 `.env` 的值，第二參數是預設值
- **設定快取** — `php artisan config:cache` 快取設定，`php artisan config:clear` 清除
- **Docker Compose 一鍵啟動** — 一個 `docker compose up -d` 拉起 App + MySQL + Redis 三個容器

> **開發方式補充**：本教案以 **Docker Compose** 為主要開發環境方案。其他常見方式包括：
> - 本機直接安裝 MySQL + Redis（適合不想用 Docker 的場景）
> - [Laravel Sail](https://laravel.com/docs/sail)（Laravel 官方 Docker 方案）
> - [Laragon](https://laragon.org/)（Windows 本機環境方案）

#### ④ 動手做

**必做**
1. 執行 `docker compose up -d`，用 `docker compose ps` 確認三個容器都在 running 狀態
2. 比對 `.env` 中的 `DB_HOST` 與 `config/database.php`，說明 `env()` 如何讀取設定

**延伸挑戰**
1. 新增一組自訂設定 `config/custom.php`，用 `config('custom.foo')` 讀取

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| 改了 `.env` 但沒生效 | 設定被快取 | `php artisan config:clear` 清除快取 |
| `docker compose up` 後 App 連不上 DB | MySQL 容器尚未 ready | 等 healthcheck 通過後重試；或查看 `docker compose logs db` 確認狀態 |
| `.env` 變數格式錯誤 | 值包含空格或特殊字元未加引號 | 用雙引號包住值，如 `APP_NAME="My App"` |

#### ⑥ 驗收標準（DoD）

能解釋 `.env` 與 `config/` 的差異，並能成功用 Docker Compose 啟動完整環境。

### U05｜Migration 與資料庫設計（90 min）

#### ① 為什麼先教這個？

U02-U03 用的是假資料，接下來要換成真的。第一步是「設計資料庫結構」——用 Laravel Migration 管理 schema 版本，確保多人協作時每個人的 DB 長一樣。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `database/migrations/xxxx_create_books_table.php` | 建表 Migration |
| `database/seeders/BookSeeder.php` | 塞入範例資料（seed data） |
| `database/seeders/DatabaseSeeder.php` | Seeder 入口 |

#### ③ 核心觀念

- **Migration 檔案命名** — `yyyy_mm_dd_hhmmss_create_books_table.php`，時間戳確保執行順序
- **`up()` 與 `down()`** — `up()` 執行變更、`down()` 回滾變更
- **Schema Builder** — `Schema::create()` 建表、`$table->string()` 定義欄位
- **`php artisan migrate`** — 執行所有未執行的 migration
- **`php artisan migrate:rollback`** — 回滾上一次 migration
- **Seeder 塞入測試資料** — `php artisan db:seed` 執行 seeder

#### ④ 動手做

**必做**
1. 閱讀 `create_books_table.php`，說出每個欄位的設計理由
2. 執行 `php artisan migrate`，確認 books 表建立成功
3. 新增 `V3_add_publisher_column` migration，為 books 表加一個 `publisher` 欄位
4. 執行 `php artisan db:seed`，確認範例資料已塞入

**延伸挑戰**
1. 思考題：如果 migration 部署後發現欄位型別設計錯誤，該怎麼修正？（提示：新增修正用的 migration）

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Migration 執行順序錯誤 | 時間戳不對 | 確認檔名的時間戳正確；或刪除後重新建立 |
| 外鍵約束失敗 | 參照的表尚未建立 | 確保被參照的表先建立（migration 順序） |
| rollback 後 migrate 報錯 | down() 方法沒寫或寫錯 | 檢查 down() 方法邏輯 |

#### ⑥ 驗收標準（DoD）

能撰寫一支新的 migration 並在執行 `php artisan migrate` 時自動套用，且能用 `php artisan migrate:status` 確認狀態。

### U06｜Eloquent 串接資料庫（120 min）

#### ① 為什麼先教這個？

U05 設計好了資料表，這堂課要把假資料換成真的——用 Eloquent ORM 串接 MySQL，讓 Controller 直接從 DB 讀寫資料。**完成後學員會第一次看到「頁面上的資料來自真實資料庫」的完整 CRUD 循環。** 此刻先不抽 Service 層，Controller 直接操作 Model——保持最短路徑讓 CRUD 跑起來。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Models/Book.php` | Eloquent Model（對應 books 表） |
| `Http/Controllers/BookController.php` | MVC 端點（改用 Eloquent 查詢） |
| `Http/Controllers/Api/BookApiController.php` | API 端點（改用 Eloquent 查詢） |

#### ③ 核心觀念

- **Active Record 模式** — 一個 Model 物件對應一筆資料庫記錄
- **`$fillable` 與 `$guarded`** — 定義可批量賦值的欄位（Mass Assignment 防護）
- **基本 CRUD**：
  - `Book::all()` — 查詢全部
  - `Book::find($id)` — 依主鍵查詢
  - `Book::create([...])` — 新增
  - `$book->update([...])` — 更新
  - `$book->delete()` — 刪除
- **Query Builder 進階查詢** — `Book::where('author', 'xxx')->get()`
- **替換假資料** — 修改 U02/U03 的 Controller，把 hardcoded list 改成呼叫 `Book::all()`，頁面立刻顯示真實 DB 資料

> **講師提示**：這是課程的第一個「啊哈時刻」——學員親眼看到頁面從假資料切換到真實 DB 資料。建議 live demo 時先秀假資料頁面，再切換到 Eloquent 呼叫，重新整理瀏覽器的瞬間最有衝擊力。

#### ④ 動手做

**必做**
1. 閱讀 `Book.php` Model，理解 `$fillable` 的用途
2. 修改 `BookController@index`，把假資料替換為 `Book::all()`，重新整理頁面確認顯示 DB 資料
3. 實作 `BookController@store`，接收表單資料並用 `Book::create()` 新增記錄

**延伸挑戰**
1. 新增 `findByIsbn($isbn)` 方法，使用 Query Scope 封裝查詢條件

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| `MassAssignmentException` | 欄位不在 `$fillable` 中 | 將欄位加入 Model 的 `$fillable` 陣列 |
| 查詢結果是空的 | 忘了執行 seeder | `php artisan db:seed` |
| Model 找不到 | namespace 或類名錯誤 | 確認 `App\Models\Book` |

#### ⑥ 驗收標準（DoD）

能在瀏覽器看到來自 DB 的書籍清單（不再是假資料），並能新增一筆書籍記錄。

### U07｜Service 與 Resource 分層（90 min）

#### ① 為什麼先教這個？

U06 完成了 CRUD，但 Controller 直接操作 Model——所有邏輯擠在一起，改一個地方可能影響全部。這堂課把程式碼「整理乾淨」：抽出 Service 層集中業務邏輯，引入 API Resource 分離輸出格式。**重構前後功能完全一樣，但程式碼更好維護、更好測試。**

> **講師提示**：先展示重構前（Controller 直接操作 Model）和重構後（Controller → Service → Model）的對比，讓學員感受到「程式碼可以一樣能跑，但結構差很多」。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Services/BookService.php` | Service 層（業務邏輯） |
| `Http/Resources/BookResource.php` | API Resource（輸出格式轉換） |
| `Http/Resources/BookCollection.php` | 集合 Resource |

#### ③ 核心觀念

- **單一職責原則** — Controller 負責 HTTP 協議、Service 負責業務邏輯、Model 負責資料存取
- **依賴注入** — 在 Controller 建構子注入 Service
- **API Resource** — 把 Model 轉換為 JSON 回應格式，類似 DTO 但更靈活
- **`BookResource::collection()`** — 批次轉換多筆記錄
- **PRG 模式（Post-Redirect-Get）** — MVC 表單送出後用 redirect 避免使用者按 F5 重複提交

#### ④ 動手做

**必做**
1. 建立 `BookService` 類別，把 CRUD 邏輯從 Controller 移過來
2. 在 Controller 注入 `BookService`，改用 Service 方法
3. 建立 `BookResource`，在 API 回傳時使用 Resource 轉換

**延伸挑戰**
1. 畫出 Controller → Service → Model 的呼叫流程圖
2. 在 Service 加入 `findByIsbn()` 方法

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Service 無法注入 | 沒有 type hint 或 namespace 錯誤 | 確認建構子參數有正確的 type hint |
| Resource 輸出欄位名稱不對 | `toArray()` 方法的 key 名稱錯誤 | 檢查 Resource 的 `toArray()` 方法 |

#### ⑥ 驗收標準（DoD）

能說明 Service 層存在的理由，能解釋 Model 與 Resource 的職責差異，頁面和 API 行為與重構前完全一致。

### U08｜REST API 完善與錯誤處理（90 min）

#### ① 為什麼先教這個？

U07 完成了分層，但 API 還缺少正確的 HTTP 狀態碼和統一的錯誤回應。使用者送了錯誤資料、查了不存在的 ID，應該要拿到清楚的錯誤訊息而非一堆 stack trace。這堂課把 API 做到「生產品質」。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Http/Controllers/Api/BookApiController.php` | REST API 端點（加入正確的狀態碼） |
| `Http/Requests/BookRequest.php` | Form Request 驗證 |
| `bootstrap/app.php` | Exception Handler 設定 |
| `Exceptions/Handler.php`（或 app.php 的 withExceptions） | 全域錯誤處理 |

#### ③ 核心觀念

- **HTTP 狀態碼語意** — 200 OK、201 Created、204 No Content、400 Bad Request、404 Not Found、422 Unprocessable Entity、500 Internal Server Error
- **`return response()->json([...], 201)`** — 新增成功回 201 而非預設的 200
- **Form Request 驗證** — 驗證邏輯獨立於 Controller，驗證失敗自動回 422
- **`$request->validated()`** — 取得驗證通過的資料
- **Exception Handler** — 統一捕捉例外並轉換為 JSON 錯誤回應

#### ④ 動手做

**必做**
1. 建立 `BookRequest` 驗證類別，定義 title、author 等欄位的驗證規則
2. 在 API Controller 使用 `BookRequest`，確認 POST 回 201、DELETE 回 204
3. 故意送缺欄位的 JSON，觀察 422 錯誤回應格式
4. 查一筆不存在的 ID，觀察 404 錯誤回應

**延伸挑戰**
1. 自訂一個業務例外（如 `DuplicateIsbnException`），回傳 409 Conflict

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Form Request 驗證沒生效 | Controller 方法參數沒有 type hint | 確認參數類型是 `BookRequest` |
| 驗證錯誤回傳 HTML 而非 JSON | 請求沒有 `Accept: application/json` header | 確認請求 header 正確；或在 Form Request 加上 `wantsJson()` 判斷 |

#### ⑥ 驗收標準（DoD）

能用 Swagger UI 完成完整 CRUD 操作（POST 201、DELETE 204），並能觸發 422 和 404 錯誤觀察統一的錯誤回應格式。

### U09｜Redis 快取整合（90 min）

#### ① 為什麼先教這個？

資料庫查詢是 Web 應用最慢的環節。把「熱資料」放在 Redis 記憶體中，回應速度可以從毫秒降到微秒等級。這堂課在 U07 建立的 Service 層上加入快取邏輯。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `Services/BookService.php` | Cache-Aside 讀寫邏輯的實際所在 |
| `config/cache.php` | 快取設定 |
| `config/database.php` | Redis 連線設定 |

#### ③ 核心觀念

- **Cache-Aside 模式** — 讀：查快取 → miss → 查 DB → 寫快取；寫/刪：操作 DB → 驅逐快取
- **`Cache::remember()`** — Laravel 提供的快取便捷方法，自動處理 hit/miss
- **`Cache::forget()`** — 清除指定 key 的快取
- **`Cache::tags()`** — 用 tag 群組管理快取（需 Redis driver）
- **快取 key 設計** — `books:{id}`，TTL 設定
- **為什麼 update / delete 要清快取？** — 避免讀到過期資料（快取一致性）

#### ④ 動手做

**必做**
1. 追蹤 `BookService@find()` 完整流程：Redis hit → 直接回傳 / Redis miss → 查 DB → 寫回 Redis
2. 用 `redis-cli` 觀察快取 key：`KEYS books:*`（教學環境限定指令）
3. 手動刪除一個 key（`DEL books:1`），再呼叫 API 觀察 cache miss 與回寫

**延伸挑戰**
1. 呼叫 `GET /api/books/1` 兩次，比較回應時間差異
2. 將 TTL 從 10 分鐘改為 1 分鐘，觀察快取命中率變化

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| `KEYS` 指令在正式環境不能用 | `KEYS` 會阻塞 Redis | 教學環境直觀好用；正式環境改用 `SCAN` |
| 快取沒更新 | 忘了在 update/delete 時清快取 | 確保 `Cache::forget()` 有被呼叫 |
| Redis 連不上 | 設定錯誤或服務未啟動 | 檢查 `.env` 的 `REDIS_HOST` 與 Docker 容器狀態 |

#### ⑥ 驗收標準（DoD）

能用 `redis-cli` 驗證快取行為（確認 key 存在、手動刪除後觀察回源），並能說明 Cache-Aside 的讀寫流程與快取失效策略。

### U10｜測試策略與實戰（120 min）

#### ① 為什麼先教這個？

不寫測試的程式碼是「可能會動」，有測試的程式碼是「確定會動」。測試是修改程式碼後不會造成回歸錯誤的安全網——這是從「學習專案」走向「生產專案」的關鍵一步。

#### ② 對應檔案

| 檔案 | 角色 |
|------|------|
| `tests/TestCase.php` | 測試基礎類別 |
| `tests/Unit/BookServiceTest.php` | 單元測試（隔離 DB） |
| `tests/Feature/BookApiTest.php` | Feature Test（API 整合測試） |

#### ③ 核心觀念

- **測試金字塔** — 單元測試（多且快）→ Feature Test（少且慢），先追求單元測試覆蓋
- **PHPUnit / Pest**：
  - `$this->mock()` — 建立 Mock 物件
  - `$this->get()` / `$this->post()` — 發送 HTTP 請求
  - `assertStatus()` / `assertJson()` — 斷言回應
- **`RefreshDatabase` trait** — 每個測試前重建資料庫
- **`@test` 註解或 `test_` 前綴** — 標記測試方法
- **AAA 模式** — Arrange（準備測試資料）→ Act（執行被測方法）→ Assert（驗證結果）

#### ④ 動手做

**必做**
1. 執行 `php artisan test`，閱讀測試報告
2. 在 `BookServiceTest` 新增一個測試方法：`test_find_returns_null_when_not_found()`

**延伸挑戰**
1. 在 `BookApiTest` 新增測試：`test_update_returns_200()`
2. 新增 404 與 Validation fail 測試案例
3. 觀察測試速度差異：Unit Test vs Feature Test，記錄並解釋差異原因

#### ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| 測試時資料庫錯誤 | 沒有設定測試資料庫 | 在 `phpunit.xml` 設定 `DB_CONNECTION=sqlite` 和 `:memory:` |
| Mock 沒生效 | 沒有正確綁定到容器 | 使用 `$this->mock()` 或 `$this->instance()` |
| Feature Test 很慢 | 每次都重建資料庫 | 考慮使用 `DatabaseTransactions` trait |

#### ⑥ 驗收標準（DoD）

能為新方法撰寫單元測試與 Feature Test，能解釋 Unit Test vs Feature Test 的差異。建議門檻：行覆蓋率 ≥ 60%，Service 核心流程覆蓋率 ≥ 80%。

| 程式碼品質 | 20% | 命名規範、分層職責明確、無重複程式碼；符合 PSR-12 |
| 測試覆蓋 | 20% | 具備正確的 Mock 運用與測試斷言；行覆蓋率 ≥ 60%，Service 核心流程 ≥ 80% |
| 問題分析 | 20% | 能判讀錯誤 log 並說明修復思路；遇到踩坑問題能自行排查 |

### 結訓驗收專案

> **設計理念**：驗收專案採用與教案**不同的業務主題**，驗證學員是否真正理解分層架構與開發流程，而非只是照抄書籍管理系統的程式碼。

**驗收方式**：限時實作（建議 3–4 小時）

**建議主題**：「員工通訊錄管理系統」（或其他具備 CRUD 性質的主題，如待辦事項、商品庫存）

| 驗收項目 | 達成條件 |
|----------|----------|
| Migration | 至少一支建表 Migration + 一支 Seeder |
| Model + Resource | Model 對應 DB 表、Form Request 含驗證、API Resource 做輸出轉換 |
| Eloquent | 完成基本 CRUD 的資料存取 |
| Service 層 | 業務邏輯集中在 Service |
| REST API | 至少 4 支 API（GET list / GET by ID / POST / DELETE），狀態碼正確 |
| Blade 頁面 | 至少一個列表頁 + 一個新增表單頁，能正常操作 |
| 測試 | 至少 1 支 Service 單元測試 + 1 支 Feature Test |
| 全部可運行 | `php artisan serve` 能啟動、頁面能操作、API 能呼叫、`php artisan test` 全綠 |

**評分標準**：同上方課程練習評量比重。

### 標準化評語範例

| 等級 | 評語範例 |
|------|----------|
| 優秀 | 功能完整且程式碼結構清晰，測試覆蓋充分，能獨立排查問題並提出合理的設計理由 |
| 達標 | 核心功能完成，分層架構正確，有基本測試，能在提示下排查常見問題 |
| 待加強 | 部分功能缺失或分層不明確，測試不足，排查問題時需要較多協助 |
| 未達標 | 核心 CRUD 流程無法跑通，建議補做指定練習後重新驗收 |
